# プロジェクト概要

このプロジェクトは、RPGツクールMZ向けのプラグイン開発プロジェクトです。  
AIエージェントはコードの生成、修正、および提案をおこなう際、以下のルールとコンテキストを厳守してください。

## ディレクトリとファイルの役割

- `docs/mz_classes.md` : RPGツクールMZのクラスリストが記載されています。ツクールのクラス構造を理解する際のリファレンスとして使用してください。
- `docs/mz_plugin_pitfalls.md` : プラグイン開発で踏みがちな落とし穴と前提知識をまとめたドキュメントです。実装を始める前に必ず確認してください。
- `docs/mz_architecture.md` : クラス間の呼び出し構造・責務や、バトルの流れやメッセージウィンドウの処理順など、クラス間の関係と内部処理をまとめたドキュメントです。
- `mz_scripts/` : ディレクトリの直下に、RPGツクールMZのコアとなるソースファイル（`rmmz_*.js` など）が配置されています。 `mz_scripts/plugins/` 以下には、参考となるプラグインが格納されています。 `mz_scripts/` 以下のファイルを直接変更することは**絶対に禁止**します。参照のみに使用してください。
- `my_plugins/` : 開発中のオリジナルプラグインを格納するディレクトリです。新しいプラグインを作成する場合は、すべてこのディレクトリ内に配置してください。プラグインは `my_plugins/PKRN_xxx/` のように、機能ごとにサブディレクトリを作成して管理してください。
  - サブディレクトリ内には、プラグインのJSファイルと `README.md` を配置します。
  - `README.md` の書き方は `my_plugins/PKRN_TPNoRegenState/README.md` や `my_plugins/PKRN_SmartCharm/README.md` を参考にしてください。 `README.md` の改行には行末にスペース2つが必要なことを忘れずに。

## ファイル命名規則と作成ルール

- 新規プラグインのファイル名は `PKRN_xxx.js` のフォーマットで作成する（例： `PKRN_SaveMax.js` ）
- ファイルの先頭には、必ず以下の著作権とライセンスのコメントブロックを配置すること。年（YYYY）には作成した年を入れる（継続して更新していく場合は作成年-最終更新年といった表記にする）

```javascript
// --------------------------------------------------------------------------
//
// [Plugin Name]
//
// Copyright (c) YYYY hatonekoe
// This software is released under the MIT License.
// https://opensource.org/license/mit
//
// --------------------------------------------------------------------------
```

## コーディング規約 (プラグインの実装方法)

1. **即時実行関数 (IIFE) と Strict モード**
   - ファイル内の処理は必ず即時実行関数 `(() => { 'use strict'; ... })();` （または `(function() { 'use strict'; ... })();`）でラップし、スコープの先頭で `'use strict';` を宣言してグローバルスコープを汚染しないようにする
2. **アノテーションの記述**
   - 英語の `/*:` と 日本語の `/*:ja` の2つのヘルプ・パラメータ記述ブロックを必ず用意する。ターゲットとしては `@target MZ` (RPGツクールMVにも対応している場合は `@target MV MZ` ) を用いる
   - パラメータ等の詳細な記法やプラグインコマンドの定義、構造体の書き方については、以下の形式に準拠する
     - **基本的なパラメータ**: `@param パラメータ名`、`@text 表示名`、`@desc 説明`、`@default 初期値`、`@type 型`を使用する。型がリストや構造体の場合は `@type struct<構造体名>[]` などを指定する
     - **プラグインコマンド**: `@command コマンド名`、`@text 表示名`、`@desc 説明` に続け、各引数ごとに `@arg 引数名`、`@text 表示名`、`@desc 説明`、`@default 初期値`、`@type 型` などを記述する
     - **構造体定義**: 構造体パラメータを使用する際は、メインのブロックとは別に `/*~struct~構造体名:`（日本語用は `/*~struct~構造体名:ja`）のブロックを作成し、その中で個々の `@param` や `@text` などを定義する
   - **パラメータ名の命名規則**
     - プラグインパラメータ名と、それに対応するステートメモタグの識別子部分は一致させる（例：パラメータ名 `StunRate` → タグ名 `SmartCharm_StunRate`）
     - ステートのメモタグで上書きされるプラグインパラメータという理由で `Default` を名前に付けない。`StunRate` のように機能そのものを表す名前にする
     - `select` 型パラメータの `@value` には camelCase を使用する
     - `select` 型パラメータの `@option` 並び順は、`@default` と同じ値を先頭に置く
3. **既存メソッドの拡張**
   - ツクールの既存クラス（例: `Window_Base` 等）のメソッドを書き換える場合は、可能な限り「エイリアス」を利用する
   - 以下の例のように、この場合はメソッドコメントを、メソッドのすぐ上ではなく、エイリアスの定義の上に記述すること。
   - 既存クラスに追加する独自プロパティ名は `_プラグイン機能名 + 説明` の形式にする。他プラグインとの衝突を避けるため、プラグイン固有の接頭辞を必ず付けること
   - 例: 
     ```javascript
        /**
         * バトル終了時に独自プロパティをリセットする
        */
        const _Game_Battler_onBattleEnd = Game_Battler.prototype.onBattleEnd;
        Game_Battler.prototype.onBattleEnd = function() {
            _Game_Battler_onBattleEnd.call(this);
            this._pluginNameSomePropertyName = false;
        };
     ```
4. **パラメータの受け取り**
   - プラグインパラメータを受け取る際は `PluginManager.parameters("PKRN_xxx")` を使用し、必要に応じて型変換をおこなう
5. **Linter (構文チェッカー) 的なルールの遵守**
   - 厳格モード (`'use strict';`) を前提とし、未宣言の変数の使用、暗黙のグローバル変数の作成を徹底して避ける
   - 変数宣言には可能な限り `const` と `let` を用い、`var` の使用は原則禁止
   - `==` や `!=` ではなく、厳密等価演算子 (`===` や `!==`) を使用する。ただし `== null` や `!= null` を用いて `null` と `undefined` の両方をチェックする場合は例外とする
   - 条件式における暗黙的なBoolean型への型変換は避け、空配列や null 等のチェックは明示的におこなう
6. **コメントのルール**
   - コードの説明は日本語のコメントでおこなう。簡潔にすぐ理解できる文章で、「です・ます」の表現は排し、末尾の句点も不要
   - メソッドにはJSDoc形式でのコメントを付ける。「エイリアス」を用いる場合は、メソッド自体のすぐ上ではなく、エイリアスの定義の上にコメントを記述すること
7. **.editorconfig の遵守**
    - コード生成・修正時は、必ずリポジトリ直下の `.editorconfig` を参照し、設定に従うこと
    - 生成した内容が `.editorconfig` に反していないかを自己点検する

## 情報収集に関する指示

特定の機能の処理の流れや変数名を調べる必要があるときは、推測でコードを書かずに `grep_search` 等を用いて必ず `mz_scripts/` 内のコアスクリプトを検索・参照してから実装をおこなう

## ドキュメントの更新に関する指示

実装を通じてツクールMZの内部動作について新たな知見が得られた場合、以下のルールに従って `docs/` 以下のドキュメントを更新すること。

### 更新するタイミング

以下のいずれかに該当する知見が得られたとき、実装完了後にドキュメントへ書き足す。  
ユーザーに確認を取る必要はなく、エージェントが判断して記述してよい。

- コアスクリプトを読んで初めてわかった**メソッドの呼び出し順序・タイミング**
  （例：「○○メソッドはバトル外でも呼ばれる」「○○はセーブロード後に呼ばれない」）
- **落とし穴**になりうる挙動（知らないとバグを踏む、知っていれば最初から避けられるもの）
- **クラス間の関係・処理フロー**で、コードを読まないと把握しにくいもの

単なる実装の詳細（「このプラグインのこの変数はこう動く」）は書かなくてよい。  
**別のプラグインを作るときにも役立つかどうか**を基準にすること。

### どのファイルに書くか

| 内容 | ファイル |
|---|---|
| 落とし穴・初期化タイミング・意外な挙動など | `docs/mz_plugin_pitfalls.md` |
| クラスの継承関係・処理フロー・フックポイント | `docs/mz_architecture.md` |

**両ファイルの責務の区別：**

- `mz_architecture.md` には「こういう処理フローになっている」という**構造の記述**のみを書く。注意喚起や「こうしないとバグになる」といった内容は含めない。
- `mz_plugin_pitfalls.md` には「こうしないとバグになる」「この挙動を知らないと失敗する」といった**注意喚起**を書く。
- 迷ったら、その知識が「読み物として理解を深めるもの」なら architecture、「実装時に参照するチェックリスト」なら pitfalls に書く。
